# 12 — 体验目标与对标基线

> 文档索引：[README.md](../README.md) · 关联：全部分章（本文是横切层）
> 定位：各分章回答「怎么做」；本文回答「**做到什么程度才算好**」——每个体验维度的对标对象、可测量的达标线（P0）与超越点（P1）、诚实声明的不及之处、以及反目标（明确不做的事）。所有量化指标带编号（如 `OP-2`），供回归集引用。

> 本章的外部产品描述来自设计阶段的对标记录，本轮没有逐项复测上游当前版本，公开引用前均**待确认**。表中数字是目标而非现状；当前验证边界见 §8。

## 1. 对话体验 —— 对标 Codex Desktop / Claude Code Desktop

**为什么对标它们**：终端/桌面 Agent 产品把「流式文本 + 工具活动流 + 审批卡片」三者混排的信息密度做到了当前最好；网页聊天产品（ChatGPT/Claude 官网）只是纯对话基线，缺工具活动语言。

### 1.1 达标线（P0）

| ID   | 指标                 | 目标                                                                                                                                | 测量方法            |
| ---- | -------------------- | ----------------------------------------------------------------------------------------------------------------------------------- | ------------------- |
| CH-1 | 发送响应             | 按下发送 → 用户消息上屏 + 思考态出现 ≤ **100ms**（本地乐观渲染，不等引擎 ack；ack 失败回滚并 toast）                                | e2e 计时            |
| CH-2 | 流式渲染             | 首 token 到屏 = 网络延迟 + ≤ 50ms 引擎处理；增量渲染稳定 60fps（16ms 合帧已定于 01 §3.5）                                           | perf 埋点           |
| CH-3 | 中断生效             | Esc → 流停止 + UI 进入已中断态 ≤ **200ms**（AbortSignal 直达 fetch 与工具）                                                         | e2e 计时            |
| CH-4 | 工具活动可读性       | 任意时刻用户 1 秒内能答出「Agent 正在干什么、第几步」：运行中卡片永远可见（折叠组自动展开尾部一张，09 §4.2），组头实时显示 ✓/✗ 计数 | 可用性测试 5/5 通过 |
| CH-5 | 插话体验             | 运行中输入框始终可打字；Enter 后立即显示「已插话，将在下次模型调用前生效」胶囊，实际注入后消息流出现插话标记                        | e2e 断言            |
| CH-6 | 分支切换             | ‹n/m› 切换渲染 ≤ **100ms**（纯本地树查询）；切换后视口锚定在分叉点消息                                                              | e2e 计时            |
| CH-7 | 会话恢复             | 重开侧边栏/全屏页 → 完整会话渲染 ≤ **500ms**（p95，≤1000 节点会话，snapshot 一次到位）                                              | perf 埋点           |
| CH-8 | 流式 Markdown 稳定性 | 代码块/表格/KaTeX/Mermaid 全支持；流式过程中**零**闪烁、零渲染报错（09 §4.1 未闭合暂缓规则）                                        | 回归集视觉断言      |

### 1.2 超越点（P1，对标对象没有的）

- **双形态同会话**：侧边栏 ⇄ 全屏页无缝互切（`Cmd/Ctrl+E`），Codex Desktop 无伴随浏览的形态；
- **队列可视化编辑**：排队消息胶囊可点开删改（Codex CLI 队列不可见不可改）；
- **插话降级透明**：不可插话轮自动降级排队并说明原因，而非静默失败。

### 1.3 反目标

- 不做打字机逐字动画（到多少渲染多少）；不做装饰性过场动画（09 §1 已定）；
- 不追求「回复速度」本身——那是模型与端点的事，Panelot 只保证不在模型之外增加可感知延迟。

## 2. 浏览器操作体验 —— 对标 Playwright MCP（能力）＋人类操作（干扰度）

**双基准**：能力上限对标 Playwright MCP（当前 Agent 操作浏览器的可靠性天花板）；干扰下限对标「用户自己动手」——Agent 操作不该比人手更打扰浏览器。

### 2.1 达标线（P0）

| ID    | 指标         | 目标                                                                                                                                     | 测量方法   |
| ----- | ------------ | ---------------------------------------------------------------------------------------------------------------------------------------- | ---------- |
| OP-1  | 感知覆盖     | 回归集页面上，L1 快照的可交互元素覆盖率 ≥ **95%**（以 Playwright MCP snapshot 同页产出为分母）；漏检由 05 §1.2 recall 优先规则兜底       | 回归集对比 |
| OP-2  | 一次成功率   | click/type/select 首次执行成功率 ≥ **90%**（回归集 20 场景）                                                                             | 回归集     |
| OP-3  | 自纠总成功率 | 含 stale-ref → 重新快照 → 重试路径后 ≥ **98%**                                                                                           | 回归集     |
| OP-4  | 动作延迟     | 单动作端到端（工具调用发出 → 结果 + 增量快照返回）p50 ≤ **1.2s** / p95 ≤ 6s（含 05 §4 稳定化等待）                                       | perf 埋点  |
| OP-5  | 表单效率     | 5 字段表单 ≤ **2 次** LLM 往返（read_page + batch_actions）                                                                              | 回归集     |
| OP-6  | 快照生成     | 普通页面 p95 ≤ **300ms**，重度 SPA ≤ 1s；单快照 ≤ 3000 tokens（05 §1.3）                                                                 | perf 埋点  |
| OP-7  | 干扰最小化   | L1 全程无浏览器级横幅；L2 attach 时长占任务时长 < **10%**（按需 attach、用完即 detach）                                                  | 埋点统计   |
| OP-8  | 登录态复用   | 用户已登录的站点直接操作，**0 配置 0 重新登录**（Playwright 独立 profile 做不到，这是核心差异化）                                        | e2e        |
| OP-9  | 人机协作     | 用户在页面手动输入（isTrusted 事件）→ 自动暂停 ≤ **300ms**；恢复时 Agent 强制重新快照感知人为变化                                        | e2e        |
| OP-10 | 降级不死循环 | 感知三级降级链（L1→AXTree→截图）每级失败必有显式错误，同一失败路径重试 ≤ 2 次即换级或向用户说明并提问（nanobrowser #126 反例为回归样本） | 回归集     |

### 2.2 超越点（P1，Playwright MCP 做不到的）

零安装（无 Node、无 CDP 端口、无独立浏览器实例）· 日常浏览器登录态与 Cookie 天然可用 · 用户在环审批与实时可视化（元素高亮 + 浮动指示器）· 人机随时交接。

### 2.3 诚实声明的不及之处（各有兜底路径）

| 差距                                            | 兜底                                                                          |
| ----------------------------------------------- | ----------------------------------------------------------------------------- |
| L1 合成事件可能被反自动化框架忽略（非 trusted） | 目标是升级到 L2 trusted 事件；当前仅部分工具有 CDP 路径，不能视为通用自动升级 |
| 跨域 iframe / closed shadow root L1 不可达      | L2 `pierce` 是目标兜底；当前尚未接入                                          |
| 无网络层拦截/mock（Playwright 强项）            | 非目标场景，不追                                                              |
| 无多浏览器（Firefox/WebKit）                    | Chrome/Edge only，明确非目标                                                  |

### 2.4 反目标

不做无头批量自动化与爬虫场景（Playwright 的主场）；不做录制回放；不为提高成功率而默认全程 attach debugger（nanobrowser 式体验倒退）。

## 3. Provider / 配置体验 —— 对标 OpenWebUI（补短板）

**为什么**：设计阶段选择 OpenWebUI 作为 BYOK 兼容面的参考，并把并发拉模型、自定义头和多 key 设为 Panelot 的目标能力。这里不把历史对标结论当作上游当前事实；Panelot 的实际实现边界见 03 §1/§6。

| ID   | 指标         | 目标                                                                                                                     | 测量方法       |
| ---- | ------------ | ------------------------------------------------------------------------------------------------------------------------ | -------------- |
| PV-1 | 配置速度     | 从打开设置到发出第一条消息 ≤ **60s**（预置模板一键填充，只输 key，内联 Verify）                                          | 可用性测试     |
| PV-2 | Verify 归因  | 四类错误（key 无效/不可达/需 host 权限/协议不符）100% 给出人话结论与下一步指引，零裸报错                                 | 回归集         |
| PV-3 | 模型拉取     | 全连接并发、单连接 4s 超时独立标红，任一失败不阻塞选择器；缓存后打开选择器 ≤ **200ms**                                   | perf 埋点      |
| PV-4 | 端点兼容     | 兼容回归表（官方双家 + OpenRouter/DeepSeek/Moonshot/智谱/百炼/Ollama/LM Studio/一个自建中转，≥10 端点）流式+工具调用全绿 | 兼容回归表维护 |
| PV-5 | Key failover | 多 key 轮换/失效切换对用户无感（对话不中断，仅设置页标记失效 key）                                                       | 故障注入测试   |

反目标：不做模型市场/排行榜；不内置任何官方 key 或中转（零后端原则）。

## 4. 审批与安全体验 —— 对标 Codex CLI approvals

**目标一句话**：安全是硬闸（06 的两轴与 Gatekeeper 不为体验让步），体验目标是让「必要的打扰」次数最少、单次成本最低。

| ID   | 指标       | 目标                                                                                                                                     | 测量方法   |
| ---- | ---------- | ---------------------------------------------------------------------------------------------------------------------------------------- | ---------- |
| AP-1 | 理解成本   | 不看文档能懂：卡片含 element 人话描述 + 完整参数 + 目标站点；可用性测试 5/5 受试者能答出「批准后会发生什么」                             | 可用性测试 |
| AP-2 | 打扰预算   | 操作询问档跑「三站比价」基准任务，全程审批 ≤ **3 次**（读类零审批 + 显式会话授权）                                                       | 回归集     |
| AP-3 | 响应速度   | 审批卡片出现 → 焦点就位可按 Y/S/A/N ≤ **100ms**；全键盘完成，零鼠标                                                                      | e2e        |
| AP-4 | 离场审批   | 无 UI 时：系统通知 → 点击 → 直达待审批卡片，≤ **2 跳**；5 分钟超时按拒绝并留痕                                                           | e2e        |
| AP-5 | 注入拦截   | 注入攻击回归集（伪造 system 指令/伪造审批文案/诱导外域/诱导 run_javascript，10 §8）硬闸层拦截率 **100%**（提示词层允许失守，硬闸不允许） | 回归集     |
| AP-6 | 拒绝可恢复 | decline 附言回填模型后，模型换路或收敛提问的比例 ≥ 80%（不原样重试被拒动作）                                                             | 回归集     |

反目标：绝不提供「自动批准一切」的默认档换流畅（`never` = 拒绝的语义已在协议层写死）；审批 UI 绝不出现在网页内。

## 6. 性能与可靠性 —— 对标原生应用的「不可见基建」

**目标一句话**：MV3 SW 的死活、IndexedDB 的读写，用户全程无感知。

| ID   | 指标        | 目标                                                                                          | 测量方法        |
| ---- | ----------- | --------------------------------------------------------------------------------------------- | --------------- |
| RL-1 | SW 恢复无损 | 任意时刻杀 SW，数据损失 ≤ 1 个流式中 item（01 §4 落库规范）；「继续」一键续跑成功率 ≥ **95%** | 故障注入回归    |
| RL-2 | 长会话性能  | 2000 节点会话：滚动 60fps（虚拟滚动）、切换会话骨架屏 → 完整渲染 ≤ 800ms                      | perf 埋点       |

## 8. 自动化验证基建

- **确定性场景集**：20 个无模型 actionability 场景覆盖可见、稳定、可用、可编辑、接收事件和 DOM 连接状态；
- **当前 Playwright 范围**：persistent Chromium context 加载生产 unpacked extension，验证 MV3 Service Worker、options 页面、权限清单、快照 ref、填值回显、closed shadow root 与跨域 iframe；
- **覆盖率门禁**：使用当前仓库基线作为只能上调的棘轮，并对 Gatekeeper 等核心文件设置更高的独立阈值。

### 8.1 验证状态

当前自动化测试覆盖若干**机制不变量**：流式 Markdown fence、snapshot/ref 过期与受限恢复、actionability、结构化动作失败、快照 diff、自纠路径、注入样本的 Gatekeeper 硬闸、引擎重建 snapshot 和附件配额。它们不能证明 CH/OP/AP/RL 表中的百分比、延迟或真实站点成功率已经达标。

仍需自动化覆盖：

- AP-4 已接入待审批/恢复通知并点击直达对应会话；通知到卡片的跳数与 5 分钟超时仍需扩展 E2E 验证；
- Plugin 安装前的独立确认页与内置精选内容；当前安装器会原子校验/写入并在安装后展示完整资产清单；
- i18n：对话核心组件已 key 化，设置页文案尚未收编。

## 9. 指标校准约定

本章的量化数字（OP-2 的 90%、AP-2 的 3 次等）是设计初值，非实测值。回归集建成后的校准约定：

- 首轮回归实测后把各数字校准一次并版本化，此后随大版本复核；
- OP 系列按模型档位分档（如「旗舰 90% / 中档 75%」），避免体验目标绑死单一模型；
- 默认权限规则集（AP-2 打扰预算与 AP-5 安全红线的平衡点）用真实任务数据定，出厂前从严；
- 基准场景优先跑本地 fixture 镜像；镜像维护成本超出收益的场景改跑真实站点并接受漂移。
