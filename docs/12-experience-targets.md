# 12 — 体验目标与对标基线

> 上级文档：[DESIGN.md](../DESIGN.md) · 关联：全部分章（本文是横切层）
> 定位：各分章回答「怎么做」；本文回答「**做到什么程度才算好**」——每个体验维度的对标对象、可测量的达标线（P0）与超越点（P1）、诚实声明的不及之处、以及反目标（明确不做的事）。所有量化指标带编号（如 `OP-2`），供里程碑验收与回归集引用。

## 0. 对标地图总纲

| # | 维度 | 对标对象 | 一句话目标 | 规格落点 |
|---|---|---|---|---|
| 1 | 对话体验 | Codex Desktop / Claude Code Desktop | 桌面级 Agent 对话质感：流式、可插话、工具活动一目了然 | 01/04/09 |

| 3 | Provider 配置 | OpenWebUI（补其短板）/ Cherry Studio（模板丰富度） | 填 Key 60 秒内开聊，兼容一切「OpenAI 兼容」端点 | 03 |
| 4 | 审批与安全 | Codex CLI approvals | 安全但不烦：读类零打扰，写类看得懂、按键快 | 06 |
| 5 | 生态扩展 | Claude Code（Skills/Plugin/MCP） | 社区资产即插即用，粘贴即导入 | 07/08 |
| 6 | 性能与可靠性 | 原生桌面应用 | MV3 SW 生命周期对用户不可见 | 01/02 |
| 7 | 上手引导 | —（自定标准） | 安装到首次有效回答 ≤ 2 分钟 | 09 |

选对标的原则：**每个维度对标该维度的最强者，而不是整体对标某一个产品**。Codex 的对话与审批最好但没有浏览器；Playwright MCP 的操作最可靠但要本地安装、无登录态；OpenWebUI 的 BYOK 最全但配置体验有硬伤。Panelot 的产品主张 = 把各维度的最高水位线组合进一个零安装的浏览器扩展。

## 1. 对话体验 —— 对标 Codex Desktop / Claude Code Desktop

**为什么对标它们**：终端/桌面 Agent 产品把「流式文本 + 工具活动流 + 审批卡片」三者混排的信息密度做到了当前最好；网页聊天产品（ChatGPT/Claude 官网）只是纯对话基线，缺工具活动语言。

### 1.1 达标线（P0）

| ID | 指标 | 目标 | 测量方法 |
|---|---|---|---|
| CH-1 | 发送响应 | 按下发送 → 用户消息上屏 + 思考态出现 ≤ **100ms**（本地乐观渲染，不等引擎 ack；ack 失败回滚并 toast） | e2e 计时 |
| CH-2 | 流式渲染 | 首 token 到屏 = 网络延迟 + ≤ 50ms 引擎处理；增量渲染稳定 60fps（16ms 合帧已定于 01 §3.5） | perf 埋点 |
| CH-3 | 中断生效 | Esc → 流停止 + UI 进入已中断态 ≤ **200ms**（AbortSignal 直达 fetch 与工具） | e2e 计时 |
| CH-4 | 工具活动可读性 | 任意时刻用户 1 秒内能答出「Agent 正在干什么、第几步」：运行中卡片永远可见（折叠组自动展开尾部一张，09 §4.2），组头实时显示 ✓/✗ 计数 | 可用性测试 5/5 通过 |
| CH-5 | 插话体验 | 运行中输入框始终可打字；Enter 后立即显示「已插话，将在下次模型调用前生效」胶囊，实际注入后消息流出现插话标记 | e2e 断言 |
| CH-6 | 分支切换 | ‹n/m› 切换渲染 ≤ **100ms**（纯本地树查询）；切换后视口锚定在分叉点消息 | e2e 计时 |
| CH-7 | 会话恢复 | 重开侧边栏/全屏页 → 完整会话渲染 ≤ **500ms**（p95，≤1000 节点会话，snapshot 一次到位） | perf 埋点 |
| CH-8 | 流式 Markdown 稳定性 | 代码块/表格/KaTeX/Mermaid 全支持；流式过程中**零**闪烁、零渲染报错（09 §4.1 未闭合暂缓规则） | 回归集视觉断言 |

### 1.2 超越点（P1，对标对象没有的）

- **双形态同会话**：侧边栏 ⇄ 全屏页无缝互切（`Cmd/Ctrl+E`），Codex Desktop 无伴随浏览的形态；
- **队列可视化编辑**：排队消息胶囊可点开删改（Codex CLI 队列不可见不可改）；
- **插话降级透明**：不可插话轮（压缩中）自动降级排队并说明原因，而非静默失败。

### 1.3 反目标

- 不做打字机逐字动画（到多少渲染多少）；不做装饰性过场动画（09 §1 已定）；
- 不追求「回复速度」本身——那是模型与端点的事，Panelot 只保证不在模型之外增加可感知延迟。

## 2. 浏览器操作体验 —— 对标 Playwright MCP（能力）＋人类操作（干扰度）

**双基准**：能力上限对标 Playwright MCP（当前 Agent 操作浏览器的可靠性天花板）；干扰下限对标「用户自己动手」——Agent 操作不该比人手更打扰浏览器。

### 2.1 达标线（P0）

| ID | 指标 | 目标 | 测量方法 |
|---|---|---|---|
| OP-1 | 感知覆盖 | 回归集页面上，L1 快照的可交互元素覆盖率 ≥ **95%**（以 Playwright MCP snapshot 同页产出为分母）；漏检由 05 §1.2 recall 优先规则兜底 | M2 回归集对比 |
| OP-2 | 一次成功率 | click/type/select 首次执行成功率 ≥ **90%**（回归集 20 场景） | 回归集 |
| OP-3 | 自纠总成功率 | 含 stale-ref → 重新快照 → 重试路径后 ≥ **98%** | 回归集 |
| OP-4 | 动作延迟 | 单动作端到端（工具调用发出 → 结果 + 增量快照返回）p50 ≤ **1.2s** / p95 ≤ 6s（含 05 §4 稳定化等待） | perf 埋点 |
| OP-5 | 表单效率 | 5 字段表单 ≤ **2 次** LLM 往返（read_page + batch_actions） | 回归集 |
| OP-6 | 快照生成 | 普通页面 p95 ≤ **300ms**，重度 SPA ≤ 1s；单快照 ≤ 3000 tokens（05 §1.3） | perf 埋点 |
| OP-7 | 干扰最小化 | L1 全程无浏览器级横幅；L2 attach 时长占任务时长 < **10%**（按需 attach、用完即 detach） | 埋点统计 |
| OP-8 | 登录态复用 | 用户已登录的站点直接操作，**0 配置 0 重新登录**（Playwright 独立 profile 做不到，这是核心差异化） | e2e |
| OP-9 | 人机协作 | 用户在页面手动输入（isTrusted 事件）→ 自动暂停 ≤ **300ms**；恢复时 Agent 强制重新快照感知人为变化 | e2e |
| OP-10 | 降级不死循环 | 感知三级降级链（L1→AXTree→截图）每级失败必有显式错误，同一失败路径重试 ≤ 2 次即换级或 ask_user（nanobrowser #126 反例为回归样本） | 回归集 |

### 2.2 超越点（P1，Playwright MCP 做不到的）

零安装（无 Node、无 CDP 端口、无独立浏览器实例）· 日常浏览器登录态与 Cookie 天然可用 · 用户在环审批与实时可视化（元素高亮 + 浮动指示器）· 人机随时交接。

### 2.3 诚实声明的不及之处（各有兜底路径）

| 差距 | 兜底 |
|---|---|
| L1 合成事件可能被反自动化框架忽略（非 trusted） | 自动升级 L2 trusted 事件（05 §2 触发器） |
| 跨域 iframe / closed shadow root L1 不可达 | L2 `pierce` 兜底 |
| 无网络层拦截/mock（Playwright 强项） | 非目标场景，不追 |
| 无多浏览器（Firefox/WebKit） | V1 Chrome/Edge，明确非目标 |

### 2.4 反目标

不做无头批量自动化与爬虫场景（Playwright 的主场）；不做录制回放（「录制 → 生成 Skill」是 V2 独立议题）；不为提高成功率而默认全程 attach debugger（nanobrowser 式体验倒退）。

## 3. Provider / 配置体验 —— 对标 OpenWebUI（补短板）

**为什么**：OpenWebUI 是 BYOK 兼容面的事实标准，但其配置体验有公认硬伤（串行拉模型、无自定义头、单 key）——Panelot 全部补齐（03 §1/§6），体验目标定在「它的兼容面 + 无它的痛点」。

| ID | 指标 | 目标 | 测量方法 |
|---|---|---|---|
| PV-1 | 配置速度 | 从打开设置到发出第一条消息 ≤ **60s**（预置模板一键填充，只输 key，内联 Verify） | 可用性测试 |
| PV-2 | Verify 归因 | 四类错误（key 无效/不可达/需 host 权限/协议不符）100% 给出人话结论与下一步指引，零裸报错 | 回归集 |
| PV-3 | 模型拉取 | 全连接并发、单连接 4s 超时独立标红，任一失败不阻塞选择器；缓存后打开选择器 ≤ **200ms** | perf 埋点 |
| PV-4 | 端点兼容 | 兼容回归表（官方双家 + OpenRouter/DeepSeek/Moonshot/智谱/百炼/Ollama/LM Studio/一个自建中转，≥10 端点）流式+工具调用全绿 | M1 起维护 |
| PV-5 | Key failover | 多 key 轮换/失效切换对用户无感（对话不中断，仅设置页标记失效 key） | 故障注入测试 |

反目标：不做模型市场/排行榜；不内置任何官方 key 或中转（零后端原则）。

## 4. 审批与安全体验 —— 对标 Codex CLI approvals

**目标一句话**：安全是硬闸（06 的两轴与 Gatekeeper 不为体验让步），体验目标是让「必要的打扰」次数最少、单次成本最低。

| ID | 指标 | 目标 | 测量方法 |
|---|---|---|---|
| AP-1 | 理解成本 | 不看文档能懂：卡片含 element 人话描述 + 完整参数 + 目标站点；可用性测试 5/5 受试者能答出「批准后会发生什么」 | 可用性测试 |
| AP-2 | 打扰预算 | on-request 档跑「三站比价」基准任务，全程审批 ≤ **3 次**（读类零审批 + 同站同工具本 turn 记忆） | 回归集 |
| AP-3 | 响应速度 | 审批卡片出现 → 焦点就位可按 Y/A/N ≤ **100ms**；全键盘完成，零鼠标 | e2e |
| AP-4 | 离场审批 | 无 UI 时：系统通知 → 点击 → 直达待审批卡片，≤ **2 跳**；5 分钟超时按拒绝并留痕 | e2e |
| AP-5 | 注入拦截 | 注入攻击回归集（伪造 system 指令/伪造审批文案/诱导外域/诱导 run_javascript，10 §8）硬闸层拦截率 **100%**（提示词层允许失守，硬闸不允许） | M2 起回归 |
| AP-6 | 拒绝可恢复 | decline 附言回填模型后，模型换路或收敛提问的比例 ≥ 80%（不原样重试被拒动作） | 回归集 |

反目标：绝不提供「自动批准一切」的默认档换流畅（`never` = 拒绝的语义已在协议层写死）；审批 UI 绝不出现在网页内。

## 5. 生态体验 —— 对标 Claude Code

| ID | 指标 | 目标 | 测量方法 |
|---|---|---|---|
| EC-1 | Skill 导入 | 社区 top-20 SKILL.md 样本直接导入成功率 ≥ **90%**；不支持项（多文件依赖）导入时明确警告而非静默残缺 | M3 样本集 |
| EC-2 | MCP 粘贴导入 | Claude Code `mcpServers` / Cursor 配置片段识别率 **100%**（url / type: http\|sse / headers.Authorization） | 单测样本集 |
| EC-3 | OAuth 接入 | 添加 OAuth MCP 服务器 → 可调用工具：≤ 3 次点击 + 1 次浏览器授权页 | e2e |
| EC-4 | 渐进披露开销 | Skills 索引常驻 prompt 开销 ≤ 30 tokens/skill；load_skill 后行为遵循率以回归集验证 | 埋点 + 回归 |
| EC-5 | Plugin 安装透明 | 安装清单页 100% 列出将写入的 Skills/MCP/规则/站点指令；plugin 建议的 allow 一律降级 ask（08 §5） | 单测 |

反目标：V1 不做社区市场与评分体系（精选列表静态托管）；不支持含可执行代码的插件（MV3 CSP 红线）。

## 6. 性能与可靠性 —— 对标原生应用的「不可见基建」

**目标一句话**：MV3 SW 的死活、IndexedDB 的读写、上下文的压缩，用户全程无感知。

| ID | 指标 | 目标 | 测量方法 |
|---|---|---|---|
| RL-1 | SW 恢复无损 | 任意时刻杀 SW，数据损失 ≤ 1 个流式中 item（01 §4 落库规范）；「继续」一键续跑成功率 ≥ **95%** | 故障注入回归 |
| RL-2 | 长会话性能 | 2000 节点会话：滚动 60fps（虚拟滚动）、切换会话骨架屏 → 完整渲染 ≤ 800ms | perf 埋点 |
| RL-3 | 内存预算 | 引擎 + 单 UI 常态 < 300MB；快照 selector_map 等内存态随 tab 关闭即释放 | 手动 profile |
| RL-4 | 压缩无感 | auto-compaction 期间 UI 显示状态条但不阻塞查看历史；压缩后继续对话无「失忆」投诉场景（trackedOps 回归样本） | 回归集 |
| RL-5 | 存储治理 | 80% 配额预警；LRU 清理绝不触及当前活跃会话；`deleting` 标记防半删状态被回放（02 §6） | 单测 + 故障注入 |

## 8. 测量基建与验收绑定

### 8.1 基建

- **基准场景集**（M2 起，与 10 §8 的 prompt 回归集合一）：20 个脚本化场景（表单/比价/搜索提取/注入攻击）× 3 档模型，e2e 用 Playwright 驱动被测扩展跑真实站点镜像（本地 fixture 站点，避免外部站点漂移）；
- **性能埋点**：dev 构建内置 perf 面板（CH/OP/RL 系列指标本地采集与导出），**不上报**（无遥测原则）；
- **可用性测试**：M2/M4 各一轮，5 人小样本，覆盖 CH-4、AP-1、OB-1/2；
- **对标基线采集**：OP-1 的分母（Playwright MCP 同页快照）、AP-2 的 Codex 审批次数，在回归集固化时实测记录一次，写死为版本化基线。

### 8.2 里程碑验收绑定

| 里程碑 | 必须达标的指标 |
|---|---|
| M1 对话核心 | CH-1/2/3/6/7/8 · PV-1/2/3/4 · OB-1/4 |
| M2 浏览器 Agent | OP-1~10 · AP-1~6 · CH-4/5 · RL-1/4 |
| M3 生态 | EC-1~5 |
| M4 打磨上架 | RL-2/3/5 · OB-2/3 · 全量指标回归一遍出跑分表 |

## 8.3 实现状态（2026-07-03）

代码层面 M1–M4 全部完成。可自动化验证的指标已有单测/e2e 覆盖（CH-8 流式围栏、OP-1/3/6 快照与自纠、AP-5 注入硬闸 100%、RL-1/5 恢复与配额）。需真实端点/浏览器/人工的指标（首答时间 OB-1、审批理解成本 AP-1、真实站点成功率 OP-2）留待人工验证轮，跑分表在那时产出。

## 9. 开放问题

- [ ] 各量化数字（OP-2 的 90%、AP-2 的 3 次等）是拍脑袋初值，回归集建成后用实测校准一次并版本化；
- [ ] 弱模型（非一线旗舰）下 OP 系列目标是否分档（如「旗舰 90% / 中档 75%」），避免体验目标绑死单一模型；
- [ ] AP-2 打扰预算与 AP-5 安全红线的张力：granular 档的默认规则集如何出厂预置，待 M2 真实任务数据；
- [ ] fixture 站点镜像的维护成本 vs 直接跑真实站点的漂移风险，M2 试点后定。
